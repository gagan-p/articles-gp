<div class="thesis">
<div class="story-label">Core Thesis</div>

The modern software industry is stratified. At the top sit applications that users see; beneath them lies the software that applications depend upon—databases, distributed storage, messaging systems, runtimes, kernels, consensus engines, cryptographic libraries, and networking frameworks. This foundational layer is where pricing power, technical leverage, and long-term strategic influence accumulate because every higher layer inherits its assumptions, interfaces, and constraints.

The central claim of this essay is not merely that India produces comparatively little foundational software. It is that foundational software emerges only when two conditions exist simultaneously: a problem that cannot economically be rented around, and an institutional home that allows the resulting technology to outlive its original builder. China acquired both. India acquired neither. Indian firms could almost always purchase an existing solution, while the country's institutions rarely retained custody of the rare foundational systems that Indian engineers did create.

The consequence is visible in the exceptions. India's strongest foundational efforts either remained the work of exceptional individuals, became captive inside a single company, or ultimately moved abroad where capital, governance, and commercial adoption could sustain them. India's missing institution is therefore not talent, nor ambition, but custody: it has not yet built the mechanisms that keep foundational software Indian once it exists.

</div>

<div class="brief">
<ul>
<li>India now has the world's largest public open-source contributor base by developer count, yet produces comparatively few globally adopted foundational software systems.</li>
<li>Its software industry is overwhelmingly export-oriented and concentrated in services and application software rather than systems software.</li>
<li>Economic value compounds at the foundational layer because every application depends upon it.</li>
<li>Companies that own foundational software consistently generate substantially greater value per employee than firms whose primary business is software services.</li>
<li>Foundational systems are typically created only when engineers encounter constraints that cannot be solved by purchasing an existing product.</li>
<li>Those systems become durable only after custody passes from an individual company to a neutral governance institution capable of attracting broad adoption.</li>
<li>China transformed domestic constraints into databases, messaging systems, service frameworks, and infrastructure projects that later entered Apache and CNCF governance.</li>
<li>India's strongest counterexamples fit a small number of recurring patterns—lone builders, re-domiciled companies, trapped internal systems, embryonic state projects, and protocol-only initiatives.</li>
<li>The practical correction is not to imitate China's history but to identify problems unique to India that foreign vendors have little incentive to solve.</li>
<li>The argument is deliberately falsifiable: by 2030 India should be able to demonstrate globally adopted foundational software retained under durable domestic custody if the thesis is wrong.</li>
</ul>
</div>

** What Is Foundational Software?

Figure [[fig:india-software-paradox]] introduced the central paradox of this essay. To understand it, we must first define *foundational software*.

Modern software is layered. Users interact with applications. Developers build them using frameworks and libraries. Beneath both lies foundational software—the operating systems, databases, messaging systems, language runtimes and other infrastructure upon which all higher layers depend. At the bottom lies hardware, the physical platform on which all software ultimately executes.

Applications create value for users. Foundational software creates value for *other software*. A successful application serves millions of users; a successful database serves thousands of applications. Improvements made at the foundational layer therefore compound across every system built above it.

Throughout this essay, *foundational software* refers to software whose primary purpose is to enable other software rather than directly serve end users. This essay is about that layer—the floor upon which modern software ecosystems are built.

** Renting the Floor

Most organizations do not build their own operating systems, databases, messaging systems or language runtimes. They rent them. Renting the floor has obvious advantages. Organizations can focus on the business problems that distinguish them instead of recreating decades of systems engineering.

For an individual organization, renting the floor is almost always the rational decision. Building foundational software is expensive, risky and rarely provides sufficient competitive advantage to justify the investment.

The question becomes more interesting when viewed collectively. What happens when an entire software ecosystem becomes exceptionally good at renting the floor but rarely builds it? What happens when a nation produces millions of software engineers and thousands of successful applications, yet remains largely dependent on foundational software developed elsewhere?

This essay is not a criticism of renting. It is an examination of what happens when everybody rents.

** India's Software Story

India built one of the world's largest software industries. Its strengths lie in software services, enterprise applications, consulting and digital public infrastructure. Foundational software followed a different path. India has capable systems work, but fewer Indian-origin projects became broad software foundations.

China's ecosystem evolved differently. Chinese firms repeatedly turned domestic scale problems into databases, messaging systems, service frameworks, storage systems and cloud-native infrastructure.

Several points matter more than the row count. BOSS is real systems work, but it is a Linux distribution: localisation and deployment of an existing foundation, not a new operating-system architecture. PARAS belongs in the same discussion as an Indian national systems-software effort to track, but not yet a broad global ecosystem.

OpenHarmony also needs precision. It should not be reduced to "China wrote a new Linux." The point is that Huawei-origin code moved into a domestic open-source foundation and became the centre of a Chinese operating-system ecosystem.

The database contrast is sharper. ShaktiDB is PostgreSQL-derived. OceanBase and TiDB are new distributed database systems built for scale problems. This difference—adaptation versus new system design—recurs across the table.

India's sparse column is not proof of weak engineers. It is evidence of where the country's software capability accumulated: mostly above the floor, while China's accumulated across much more of the floor itself.


** A Tale of Two Ecosystems

The previous sections established two observations.

1. India built one of the world's largest software industries.
2. China built a broader ecosystem of foundational software.

Neither observation explains *why*.

This essay proposes that foundational software emerges only when two conditions exist simultaneously:

- *Forcing Function* — a problem that cannot economically be solved by renting existing software.
- *Custody* — an institution capable of sustaining the resulting software beyond its original creator.

Possessing only one condition produces isolated projects. Possessing both repeatedly produces foundational software ecosystems.

The remainder of this essay develops this argument.

** The Exceptions

The previous section showed that India's contribution to globally adopted foundational software is comparatively small. That observation, however, should not be interpreted as the absence of systems engineering.

India has repeatedly produced foundational software, somehow they seem to have lost midway. We list few of these. Rather than study each project independently, this essay groups representative Indian foundational software efforts into recurring patterns. Those patterns form the basis of the explanatory model developed in the following sections.

** Representative Indian Foundational Software

The preceding sections compared software ecosystems at a national level. Such comparisons, however, hide important individual achievements. India has produced a number of technically significant software systems spanning operating systems, databases, developer infrastructure, reusable frameworks and national protocols.

The projects presented here are illustrative rather than exhaustive. They are not ranked by commercial success or technical merit. Instead, they represent different approaches to building software below—or immediately adjacent to—the application layer.

#+begin_export html
<div class="story">
<div class="story-label">Kitty — Modern Developer Infrastructure</div>

Kitty is a modern GPU-accelerated terminal emulator created by Kovid Goyal. Unlike traditional terminal emulators that rely primarily on the CPU for rendering text, Kitty uses the graphics processor to deliver significantly higher rendering performance while supporting advanced features such as ligatures, graphics, multiple layouts and remote control.

Terminal emulators form part of the daily infrastructure used by software developers, system administrators and DevOps engineers. Although largely invisible to end users, they are fundamental tools upon which software development itself depends.

Kitty demonstrates that individual Indian developers are capable of producing world-class developer infrastructure adopted by engineers across the world.

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">Frappe — A Reusable Application Platform</div>

Frappe is an open-source full-stack application framework developed by Frappe Technologies. At its core is the DocType abstraction, through which developers describe business entities once and automatically obtain data models, forms, permissions, APIs, reports and administrative interfaces.

Rather than building a single application, Frappe provides a reusable platform upon which many applications—including ERPNext—are constructed. It therefore occupies an intermediate position between application software and foundational infrastructure.

Frappe illustrates India's strength in creating reusable software platforms that substantially reduce the effort required to build enterprise applications.

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">UPI — Infrastructure Through Protocols</div>

The Unified Payments Interface (UPI), developed under the National Payments Corporation of India (NPCI), is not an application but an interoperable payments protocol. Banks, fintech companies and payment applications communicate through a common set of interfaces while remaining independently implemented.

UPI transformed digital payments in India by defining shared rules rather than prescribing a single software implementation. Hundreds of payment applications coexist because they implement the same protocol.

UPI illustrates an alternative approach to building national infrastructure: standardising interfaces instead of building a monolithic software platform.

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">PARAS — Sovereign Operating System Initiative</div>

PARAS is an indigenous operating-system initiative intended to strengthen India's sovereign computing capabilities. Like earlier efforts such as BOSS, it addresses requirements arising from government, defence and strategic sectors where long-term control over software platforms is considered important.

Unlike commercial operating systems that evolve primarily through market competition, PARAS is driven by national capability and software sovereignty objectives.

It represents India's continuing investment in systems software and operating-system technologies.

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">ShaktiDB — Building on PostgreSQL</div>

ShaktiDB is an indigenous relational database initiative developed under the Ministry of Electronics and Information Technology (MeitY). Rather than creating an entirely new database engine, it extends PostgreSQL to satisfy Indian requirements relating to sovereignty, deployment and public-sector adoption.

The project demonstrates substantial systems engineering while simultaneously illustrating a recurring approach within India's software ecosystem: adapting and extending mature open-source foundations rather than replacing them.

ShaktiDB represents an important step towards sovereign database infrastructure.

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">MinIO, YugabyteDB and Hasura — Success Beyond India</div>

Several globally respected infrastructure software companies originated with Indian founders. MinIO developed one of the world's fastest S3-compatible object storage systems. YugabyteDB created a distributed SQL database combining PostgreSQL compatibility with horizontal scalability. Hasura built a high-performance GraphQL engine that automatically exposes databases through modern APIs.

All three compete internationally in markets traditionally dominated by large infrastructure vendors. Yet all three eventually established their corporate headquarters and commercial ecosystems outside India while retaining significant engineering operations in India.

Their stories demonstrate that India can produce globally competitive foundational software companies. They also raise an important question explored later in this essay: why does long-term commercial stewardship of these projects so often migrate beyond India's borders?

</div>
#+end_export

#+begin_export html
<div class="story">
<div class="story-label">DungBeetle — Software That Never Leaves the Organisation</div>

Not all foundational software becomes an open-source project or commercial product. Large enterprises routinely develop sophisticated internal infrastructure to solve problems unique to their own operations.

DungBeetle represents this class of software. Designed to address specific enterprise requirements, it embodies substantial systems engineering but remains tightly coupled to the organisation that created it. Such software often delivers significant internal value yet never develops an external developer community or ecosystem.

Projects of this kind are rarely visible outside their parent organisations, making them an often-overlooked component of a nation's software capability.
